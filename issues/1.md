## Summary
The `Eval (Nightly)` GitHub Actions job is failing consistently because the eval runner aborts with an embedding database/provider model identifier mismatch.

## Failing job
- Run: https://github.com/norrietaylor/distillery/actions/runs/23691599491/job/69019987837

## Error (from logs)
```
Runner error: Embedding model mismatch: database was populated with 'mock-hash-4d' but the configured provider uses 'mock-hash'. Using different models would produce incompatible embeddings.
```

This causes essentially all eval scenarios to fail, and the aggregate check reports 0% pass rate.

## Suspected cause
A persisted/pre-populated embedding DB (vector store) was created using embedding model id `mock-hash-4d`, but the current CI configuration initializes an embedding provider that reports `mock-hash`. The runner treats this as incompatible and exits before scenarios run.

## Proposed fixes
1) **Auto-reset store on mismatch (CI/test-safe):** If an `EmbeddingModelMismatchError` occurs when opening the store, delete the store path and recreate it. Optionally gate behind an env var (e.g. `DISTILLERY_RESET_STORE_ON_EMBEDDING_MISMATCH=1`).
2) **Stabilize embedding model identifier:** Ensure the mock embedding provider uses a stable model id and matches whatever is stored in DB metadata.
3) **Use per-run temp store:** Configure eval runs to always use a fresh, run-specific store path (e.g. under `tmp_path`), avoiding stale caches.

## Acceptance criteria
- `pytest -m eval` no longer fails due to embedding model mismatch.
- Nightly workflow succeeds (or at least progresses to scenario execution and reports real failures if any).
