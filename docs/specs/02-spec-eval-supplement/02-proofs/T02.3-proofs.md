### src/distillery/cli.py
New helper functions added before `_cmd_eval`:

- `_load_golden_retrieval_labels(golden_dir)` — Loads `tests/eval/golden/retrieval.yaml`,
  maps `entry_index` → content fingerprint (first 100 chars) as `entry_id`, returns
  `dict[scenario_name, list[golden_label_dict]]`.

- `_build_retrieval_tool_calls(result)` — Translates `distillery_search` tool call
  responses so `id` = `content[:100]` (content fingerprint), enabling comparison
  against content-fingerprint golden labels. Returns translated `ToolCallRecord` list.

- `_apply_retrieval_scoring(results, golden_labels_by_name)` — For each result with
  matching golden labels: calls `score_retrieval()`, attaches `RetrievalMetrics` to
  `result.retrieval_metrics`, enforces MRR >= 0.7 and precision@5 >= 0.6 by appending
  to `failure_reasons` and setting `result.passed = False`.

- `_format_retrieval_metrics_text(result)` — Returns a formatted retrieval metrics
  line (`"  retrieval: MRR=X.XXX  P@5=X.XXX  R@5=X.XXX"`) or `None` if unavailable.

## Design Decisions

**Content-fingerprint matching**: Since the golden file (`retrieval.yaml`) identifies
relevant entries by `entry_index` (position in `seed_entries`) and actual DuckDB
entries receive UUID IDs at insert time, matching by ID is not possible without
exposing seeded UUIDs from the runner. Instead, a 100-character content prefix is used
as a fingerprint — both the golden labels and the tool call responses are normalized to
this fingerprint before scoring.

**Graceful degradation**: All retrieval scoring code is wrapped in try/except for
ImportError, ensuring the CLI works normally when `[eval]` or `[ragas]` extras are
not installed.

**Threshold constants**: `_MRR_THRESHOLD = 0.7`, `_PRECISION_THRESHOLD = 0.6`, and
`_RETRIEVAL_K = 5` are defined as module-level constants for easy tuning.
