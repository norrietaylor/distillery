### distillery_check_conflicts Tool

Two-pass workflow:

**First pass** (no `llm_responses`):
- Queries `find_similar` above `conflict_threshold`
- Returns `conflict_candidates` list with `entry_id`, `content_preview`, `similarity_score`,
  and `conflict_prompt` for each candidate
- `has_conflicts` is `False` (LLM responses needed)

**Second pass** (with `llm_responses`):
- Accepts `{entry_id: {is_conflict: bool, reasoning: str}}` mapping
- Delegates to `ConflictChecker.check()` with converted tuple format
- Returns final `ConflictResult` with `has_conflicts` and `conflicts` list

### distillery_store Conflict Integration

After the existing deduplication check, `_handle_store` now:
- Creates a `ConflictChecker` with `config.classification.conflict_threshold`
- Calls `find_similar` to find similar entries
- Builds prompts via `ConflictChecker.build_prompt()`
- Includes `conflict_candidates` and `conflict_message` in the response
- If conflict check raises an exception, logs warning and continues (non-fatal)
