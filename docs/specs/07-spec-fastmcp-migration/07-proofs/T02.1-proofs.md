# T02.1 Proof Summary: Migrate Core CRUD Tools to @mcp.tool() Decorators

## Task

Convert 5 core CRUD tools from the legacy manual dispatch pattern to FastMCP
`@mcp.tool()` decorated functions within `create_server()`:
- `distillery_status`
- `distillery_store`
- `distillery_get`
- `distillery_update`
- `distillery_list`

Each tool wrapper:
1. Uses typed parameters matching the current inputSchema
2. Uses a docstring as the tool description
3. Extracts `store`, `config`, `embedding_provider`, `recent_searches` from `ctx.lifespan_context`
4. Delegates to the existing `_handle_*` async function

## Changes Made

### `src/distillery/mcp/server.py`

Added 5 `@server.tool` decorated async functions inside `create_server()`, after
the `FastMCP("distillery", lifespan=lifespan)` construction and before `return server`:

- `distillery_status(ctx)` - delegates to `_handle_status`
- `distillery_store(ctx, content, entry_type, author, ...)` - delegates to `_handle_store`
- `distillery_get(ctx, entry_id)` - delegates to `_handle_get`
- `distillery_update(ctx, entry_id, ...)` - delegates to `_handle_update`
- `distillery_list(ctx, ...)` - delegates to `_handle_list`

The `noqa: F401` comment on the `Context` import (added in T01.2) is now satisfied
since `Context` is used in the tool signatures.

No changes to any `_handle_*` functions - they remain the unmodified business logic layer.

## Notes

- `ctx.lifespan_context` is the correct FastMCP property (not `ctx.request_context.lifespan_context`
  as the spec suggested; the spec's FastMCP API reference was slightly inaccurate)
- The `noqa: PLR0913` comment suppresses ruff's "too many arguments" lint rule on
  `distillery_store` (8 params) and `distillery_update` (9 params)
- All optional parameters use `None` default + conditional inclusion in the `arguments` dict
  to avoid passing `None` values to `_handle_*` functions that use `arguments.get()`
