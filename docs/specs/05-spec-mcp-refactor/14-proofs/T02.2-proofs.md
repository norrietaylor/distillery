# T02.2 Proof Summary: Add validate_limit helper and replace hardcoded limit validation

## Task
T02.2 (task board id: 14): Replace hardcoded limit-validation patterns in domain handler
modules (search.py, crud.py) with calls to the existing validate_limit() from _errors.py.

## Requirements

### R02.2.1: validate_limit() exists in _errors.py with correct signature
validate_limit(value, min_val=1, max_val=1000, default=50) already existed in
src/distillery/mcp/tools/_errors.py from T02.1. Function returns validated int or
(ToolErrorCode, str) error tuple.

### R02.2.2: All limit validation in handlers uses validate_limit()
Replaced hardcoded patterns in 4 handler functions:
- `_handle_search` (search.py): limit range 1-200, default 10
- `_handle_find_similar` (search.py): limit range 1-200, default 10
- `_handle_aggregate` (search.py): limit range 1-500, default 50
- `_handle_list` (crud.py): limit range 1-500, default 20

## Implementation Notes

- Both search.py and crud.py now import validate_limit from _errors.py
- The validate_limit() return value is checked with isinstance(result, tuple) to
  distinguish success (int) from error ((code, msg) tuple)
- Error responses use error_response(*limit_result) to unpack the tuple
- test_mcp_server.py test `test_list_invalid_limit_returns_error` updated to expect
  INVALID_PARAMS (from validate_limit) instead of VALIDATION_ERROR (the old hardcoded code)
- The old validate_type() + manual range checks are fully replaced in all 4 handler functions
