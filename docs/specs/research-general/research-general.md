# Research Report: General

> Generated by cw-research on 2026-03-24
> Codebase: distillery

## Summary

- **Distillery** is a Python 3.11+ team knowledge base system that captures, classifies, and retrieves knowledge entries via an MCP (Model Context Protocol) server integrated with Claude Code.
- Architecture follows a **Protocol-based design** with pluggable embedding providers (Jina AI, OpenAI), a DuckDB vector store with HNSW cosine similarity indexing, and a 4-stage deduplication pipeline (SKIP/MERGE/LINK/CREATE).
- The codebase is async-first, wrapping blocking DuckDB operations via `asyncio.to_thread()`, with graceful degradation at three levels: stub embedding provider for missing API keys, fallback classification (INBOX/PENDING_REVIEW) for LLM parse failures, and non-fatal dedup checks during storage.
- Quality infrastructure includes **368 tests** across 11 modules, mypy strict mode, ruff linting, and GitHub Actions CI (lint → type check → test). All tests use in-memory DuckDB for isolation.
- Notable issue: the `distillery` CLI entry point in pyproject.toml references a non-existent `distillery.cli:main` module. CI only tests Python 3.11 despite classifiers listing 3.11-3.13.

## Architecture & Patterns

### Design Patterns

- **Protocol-Based Abstraction**: `EmbeddingProvider` and `DistilleryStore` use Python `Protocol` for structural subtyping. DuckDBStore implements DistilleryStore without inheriting it. `@runtime_checkable` on DistilleryStore.
- **Factory Pattern**: `_create_embedding_provider()` in `mcp/server.py` routes to Jina, OpenAI, or Stub based on config.
- **Strategy Pattern**: `ClassificationEngine` and `DeduplicationChecker` are stateless processors with configurable thresholds.
- **Lifespan Context Manager**: `@asynccontextmanager _lifespan()` manages initialization order: embedding provider → DuckDBStore → state registration.
- **Tool Dispatcher**: Single `_call_tool()` function routes all MCP calls to specific handlers.

### Module Boundaries

```
Layer 1: Models & Config (models.py, config.py)
Layer 2: Protocol Abstractions (embedding/protocol.py, store/protocol.py)
Layer 3: Implementations (embedding/{jina,openai}.py, store/duckdb.py, mcp/_stub_embedding.py)
Layer 4: Business Logic (classification/{engine,dedup,models}.py)
Layer 5: Integration & API (mcp/server.py)
```

### Key Abstractions

- **Entry** dataclass: 12 fields with auto-generated UUID, timestamps, version counter
- **SearchResult**: Entry + cosine similarity score
- **ClassificationResult**: entry_type, confidence, status, reasoning, suggested_tags, suggested_project
- **DeduplicationResult**: action (SKIP/MERGE/LINK/CREATE), similar_entries, highest_score, reasoning

### Conventions

- **Naming**: PascalCase classes, snake_case functions, leading underscore for private, UPPER_CASE constants
- **MCP tools**: `distillery_<action>` pattern
- **Error handling**: Structured JSON responses with `error_response(code, message)` and `success_response(data)`
- **Async methods**: All store/embedding operations are async; DuckDB uses `asyncio.to_thread()` wrappers

### Deep-Dive Findings

- **State flow**: MCP server uses a mutable `_state: dict[str, Any]` container populated during lifespan. Handlers destructure state at dispatch level and receive explicit parameters (store, embedding_provider, config).
- **Error propagation**: Three-tier system — input validation returns error_response immediately; store/embedding exceptions caught by handler try/except and converted to structured JSON; dedup failures are non-fatal (entry stored regardless).
- **No circular dependencies**: Protocol-based design, `TYPE_CHECKING` blocks, and lazy imports in handlers prevent import cycles.
- **StubEmbeddingProvider fallback**: When `config.embedding.provider == ""`, returns zero vectors. Server starts without API keys. Dedup returns all entries as "similar" (useless but non-crashing).
- **Classification prompt**: `_CLASSIFY_PROMPT` template formats content for LLM classification. Response parsing strips markdown fences, clamps confidence to [0, 1], falls back to INBOX/0.0/PENDING_REVIEW on any failure.
- **Concurrent access**: No optimistic locking — last-write-wins semantics. Version field tracks edit count but does not prevent conflicts.

## Dependencies & Integrations

### External Dependencies

| Dependency | Version | Purpose |
|-----------|---------|---------|
| duckdb | >=0.8.0 | Vector storage with VSS extension & HNSW index |
| mcp | >=0.1.0 | Model Context Protocol server SDK |
| httpx | >=0.24.0 | HTTP client for embedding APIs |
| pyyaml | >=6.0 | YAML configuration parsing |

### Integration Points

**Jina AI Embeddings API** (`embedding/jina.py`):
- Endpoint: `https://api.jina.ai/v1/embeddings`
- Model: `jina-embeddings-v3` (default), Matryoshka truncation (256-1024 dims)
- Task differentiation: `retrieval.passage` (storage) vs `retrieval.query` (search)
- Auth: Bearer token via `JINA_API_KEY` env var
- Retry: 3 attempts, exponential backoff (1s → 2s → 4s)

**OpenAI Embeddings API** (`embedding/openai.py`):
- Endpoint: `https://api.openai.com/v1/embeddings`
- Model: `text-embedding-3-small` (default, 512 dims)
- Auth: Bearer token via `OPENAI_API_KEY` env var
- Retry: 3 attempts, exponential backoff (1s → 2s → 4s)
- Persistent httpx.Client (timeout=30s)

### Internal Data Flow

1. **Entry Creation**: Content → embed → DuckDB INSERT with embedding vector → dedup check (non-fatal)
2. **Search**: Query → embed (retrieval.query) → HNSW cosine similarity → filter → ranked results
3. **Classification**: Content → LLM prompt → parse JSON response → status decision (confidence threshold)
4. **Deduplication**: Content → find_similar (threshold=0.60) → score mapping → action (SKIP/MERGE/LINK/CREATE)

### Deep-Dive Findings

- **DuckDB features**: VSS extension with persistent HNSW (`hnsw_enable_experimental_persistence = true`), `array_cosine_similarity()` for scoring, `list_has_any()` for tag filtering, JSON metadata column, soft-delete via status field, 0600 file permissions.
- **Config loading chain**: Explicit path → `DISTILLERY_CONFIG` env var → `distillery.yaml` in CWD → defaults. Validation enforces `link ≤ merge ≤ skip` threshold ordering.
- **Version compatibility risks**: `mcp>=0.1.0` is very loose for a young SDK; HNSW persistence may require DuckDB ≥0.9.0 despite ≥0.8.0 constraint.
- **DuckDB concurrency**: Single shared connection, sync operations in `asyncio.to_thread()`. No true write parallelism — DuckDB is single-threaded.

### Code Quality Tools

- **Ruff**: line-length 100, rules E/W/F/I/N/UP/B/C4/SIM, ignores E501
- **MyPy**: strict mode for src/, relaxed for tests (allow untyped defs)
- **Commit convention**: Conventional Commits (feat/fix/docs/test/refactor/chore with scopes)

### Deep-Dive Findings

- **Test fixtures**: Each module defines its own fixtures locally (no conftest.py). `_make_entry()` and `_parse_response()` helpers duplicated across 5+ modules.
- **Embedding mock types**: Three tiers — `_MockEmbeddingProvider` (hash-based, 4D), `_DeterministicEmbeddingProvider` (registry + hash fallback, 4D), `_ControlledEmbeddingProvider` (registry + L2 normalization, 8D for precise threshold testing).
- **DuckDB isolation**: All tests use `:memory:` database. Per-test fixture lifecycle: create → initialize → yield → close.
- **MCP testing approach**: Handler-level testing (no full stdio protocol). All 11 tool handlers tested via direct async calls with mock arguments dicts. `run_server()` and lifespan context manager have no test coverage.
- **Coverage gaps**: Concurrent access scenarios, DB connection failures mid-operation, malformed YAML config, MCP transport errors, large payload edge cases, Unicode handling in content fields.

## Data Models & API Surface

### Data Models

**Entry** (`models.py`):
- Required: `content` (str), `entry_type` (EntryType), `source` (EntrySource), `author` (str)
- Auto-generated: `id` (UUID4), `created_at` (UTC), `updated_at` (UTC), `version` (int, starts 1)
- Optional: `project` (str|None), `tags` (list[str]), `status` (EntryStatus, default ACTIVE), `metadata` (dict[str, Any])
- Serialization: `to_dict()` / `from_dict()` with ISO 8601 timestamps and enum string values
- Immutable fields: `id`, `created_at`, `source`

**Enums**:
- `EntryType`: session, bookmark, minutes, meeting, reference, idea, inbox
- `EntrySource`: claude-code, manual, import
- `EntryStatus`: active, pending_review, archived
- `DeduplicationAction`: skip, merge, link, create

### API Endpoints

11 MCP tools exposed via stdio transport:

| Tool | Operation | Required Input | Key Output |
|------|-----------|---------------|------------|
| `distillery_status` | Read | None | DB stats, entry counts, model info |
| `distillery_store` | Create | content, entry_type, author | entry_id, dedup warnings |
| `distillery_get` | Read | entry_id | Full entry or NOT_FOUND |
| `distillery_update` | Update | entry_id + updates | Updated entry (version++) |
| `distillery_search` | Search | query, [filters] | Entries ranked by cosine similarity |
| `distillery_find_similar` | Search | content, [threshold] | Similar entries >= threshold |
| `distillery_list` | Read | [filters, limit, offset] | Entries by created_at DESC |
| `distillery_classify` | Update | entry_id, entry_type, confidence | Entry with classification metadata |
| `distillery_review_queue` | Read | [limit, entry_type] | Pending entries with preview |
| `distillery_resolve_review` | Update | entry_id, action | Approved/reclassified/archived entry |
| `distillery_check_dedup` | Analysis | content | action, score, similar_entries |

### Request/Response Shapes

**Error format**: `{"error": true, "code": "...", "message": "...", "details": {...}}`
**Error codes**: INVALID_INPUT, NOT_FOUND, STORE_ERROR, STATUS_ERROR, SEARCH_ERROR, SERVER_ERROR

**Dedup thresholds**: skip=0.95, merge=0.80, link=0.60 (configurable, must satisfy link ≤ merge ≤ skip)

### Deep-Dive Findings

- **DuckDB schema**: `entries` table with `embedding FLOAT[{dimensions}]` (dynamic), `metadata JSON`, `tags VARCHAR[]`. HNSW index with cosine metric. `_meta` table stores `embedding_model` and `embedding_dimensions` to prevent mixed-model databases.
- **Filter SQL**: `_build_filter_clauses()` produces parameterized SQL. `entry_type` supports single or multi-value (`IN` clause). Tags use `list_has_any()`. Dates use `>=`/`<=` on `created_at`.
- **Review workflow**: classify → confidence check → ACTIVE or PENDING_REVIEW → review_queue (list pending) → resolve_review (approve/reclassify/archive). Metadata tracks `classified_at`, `classification_reasoning`, `reviewed_at`, `reviewed_by`, `reclassified_from`.
- **Store tool dedup threshold**: Default 0.92 (higher than merge threshold 0.80), so only near-duplicates trigger warnings during storage.
- **Versioning**: Integer counter incremented on every update. No optimistic locking — concurrent updates use last-write-wins.
